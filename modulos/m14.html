O Módulo Transversal 14 — Feature Engineering e Feature Store tem como finalidade especificar, construir, documentar, validar, armazenar, versionar, reutilizar e monitorar atributos analíticos, também denominados features, produzidos a partir dos dados processados no modelo de referência UnespDataLens-RM.

1. Finalidade do módulo

O Módulo Transversal 14 — Feature Engineering e Feature Store tem como finalidade especificar, construir, documentar, validar, armazenar, versionar, reutilizar e monitorar atributos analíticos, também denominados features, produzidos a partir dos dados processados no modelo de referência UnespDataLens-RM.

Esse módulo atua de forma transversal porque a criação de features pode ocorrer em diferentes momentos do ciclo de dados. As features podem ser derivadas de fontes originais, dados integrados, dados tratados, dados validados, representações tabulares, séries temporais, grafos, indicadores, data marts, produtos de dados ou registros de consumo. Portanto, o módulo não pertence apenas a uma etapa linear do pipeline, mas se conecta a praticamente todos os módulos do UnespDataLens-RM.

No contexto da Engenharia Analítica de Dados, feature engineering corresponde ao processo de transformar dados disponíveis em atributos mais informativos, úteis e adequados para análises, modelos preditivos, indicadores, dashboards, sistemas de recomendação, análise de risco, agrupamentos, explicabilidade, tomada de decisão e agentes inteligentes. Esse processo envolve selecionar, construir, combinar, transformar, agregar, normalizar, codificar, temporalizar, contextualizar ou representar variáveis de modo que elas capturem melhor o fenômeno investigado.

A feature store, por sua vez, corresponde a uma estrutura organizada para armazenar, documentar, versionar, controlar, reutilizar e disponibilizar features de forma consistente. Ela permite que atributos criados em um projeto não fiquem dispersos em scripts, notebooks, planilhas ou pipelines isolados, mas sejam tratados como ativos analíticos reutilizáveis, governados, rastreáveis e monitoráveis.

No UnespDataLens-RM, esse módulo é essencial porque grande parte do valor analítico dos dados depende da qualidade das features construídas. Um modelo preditivo, um painel de indicadores ou uma análise exploratória podem ser fortemente influenciados por como as variáveis foram definidas, calculadas, agregadas, temporalizadas e disponibilizadas. Features mal definidas, sem documentação, sem versionamento ou sem controle de qualidade podem comprometer resultados, gerar vieses, dificultar reprodutibilidade e reduzir confiança nos produtos analíticos.

Assim, o Módulo 14 formaliza feature engineering e feature store como componentes transversais da Engenharia Analítica de Dados, garantindo que atributos analíticos sejam produzidos, avaliados e reutilizados de forma sistemática, documentada, rastreável e adequada ao uso.

2. Objetivo geral

O objetivo geral do Módulo 14 é formalizar o processo de criação, documentação, validação, armazenamento, versionamento, reutilização e monitoramento de features no UnespDataLens-RM, garantindo que atributos analíticos sejam semanticamente coerentes, tecnicamente corretos, rastreáveis, governados, reprodutíveis e adequados para análises, modelos, dashboards, indicadores, grafos, APIs e agentes inteligentes.

3. Objetivos específicos

São objetivos específicos do Módulo 14:

  1. identificar atributos candidatos à criação de features a partir de fontes, datasets tratados, data marts, representações analíticas e produtos de dados;
  2. definir regras formais para criação de features derivadas, agregadas, temporais, relacionais, estatísticas, categóricas, textuais, semânticas ou baseadas em grafos;
  3. documentar origem, fórmula, granularidade, janela temporal, atributos de entrada, regra de cálculo, versão, finalidade e limitações de cada feature;
  4. avaliar a qualidade das features quanto à completude, validade, estabilidade, distribuição, correlação, redundância, vazamento de informação, sensibilidade e risco de viés;
  5. armazenar features em uma estrutura organizada, reutilizável, versionada e controlada;
  6. diferenciar features exploratórias, features aprovadas, features depreciadas, features sensíveis, features restritas e features prontas para produção;
  7. evitar duplicação de atributos derivados e inconsistências entre cálculos realizados por diferentes equipes, scripts ou projetos;
  8. garantir consistência entre features usadas em treinamento, validação, testes, dashboards, APIs e aplicações de consumo;
  9. integrar features aos módulos de metadados, governança, linhagem, versionamento, representações analíticas, monitoramento e agentes inteligentes;
  10. produzir artefatos formais que permitam auditoria, reprodutibilidade, reuso e avaliação da contribuição das features para análises e modelos.

4. Fundamentação conceitual do módulo

Features são atributos analíticos utilizados para representar aspectos relevantes de uma unidade de análise. Elas podem ser atributos originais, atributos derivados, indicadores, agregações, transformações, medidas temporais, métricas relacionais, embeddings, variáveis categorizadas, estatísticas por janela, scores, sinais de comportamento ou representações computacionais extraídas dos dados.

No UnespDataLens-RM, uma feature deve ser tratada como um ativo analítico, e não apenas como uma coluna criada em um script. Isso significa que cada feature deve possuir definição, finalidade, origem, regra de cálculo, granularidade, versão, qualidade, restrição, sensibilidade, linhagem e condição de uso.

As features podem assumir diferentes tipos:

A feature store tem como função organizar essas features em uma estrutura reutilizável. Ela pode ser simples, como um catálogo estruturado de features com tabelas versionadas, ou mais sofisticada, envolvendo repositórios online e offline, APIs, pipelines automatizados, validação, monitoramento e integração com modelos.

No contexto do UnespDataLens-RM, a feature store pode cumprir funções como:

  • armazenar features validadas;
  • documentar regras de cálculo;
  • controlar versões;
  • registrar linhagem;
  • evitar duplicidade de features;
  • garantir consistência entre treinamento e consumo;
  • apoiar reuso em diferentes projetos;
  • controlar acesso a features sensíveis;
  • monitorar estabilidade e drift das features;
  • fornecer insumos para modelos, dashboards, indicadores e agentes inteligentes.

À luz da Design Science Research, o Módulo 14 formaliza feature engineering e feature store como artefatos técnico-metodológicos voltados a resolver problemas recorrentes em pipelines analíticos: criação dispersa de variáveis, ausência de documentação, baixa reprodutibilidade, duplicação de cálculos, inconsistência entre ambientes, vazamento de informação, falta de controle de qualidade e dificuldade de reuso.

5. Natureza transversal do módulo

O Módulo 14 é transversal porque features podem ser produzidas, consumidas, avaliadas e monitoradas em diferentes pontos do UnespDataLens-RM.

No Módulo 1 — Fontes e Inventário de Dados, identifica atributos candidatos e unidades de análise relevantes.

No Módulo 2 — Extração, Ingestão e Armazenamento Bruto, utiliza dados brutos e snapshots como base para features reprocessáveis.

No Módulo 3 — Integração e Harmonização, utiliza entidades integradas, chaves e relações entre fontes para criar features combinadas.

No Módulo 4 — Transformação, Limpeza e Enriquecimento, recebe atributos tratados, padronizados e enriquecidos, além de produzir atributos derivados.

No Módulo 5 — Qualidade e Validação de Dados, utiliza métricas de qualidade para aprovar, rejeitar ou sinalizar features.

No Módulo 6 — Desbalanceamento, Vieses e Equidade, avalia se features afetam grupos de forma desigual, se funcionam como proxies sensíveis ou se introduzem riscos de viés.

No Módulo 7 — Armazenamento Analítico, utiliza data marts, views e datasets persistidos para armazenar ou materializar features.

No Módulo 8 — Disponibilização e Consumo Analítico, disponibiliza features para dashboards, APIs, modelos, produtos de dados e agentes inteligentes.

No Módulo 9 — Metadados, Catálogo e Semântica, registra definições, descrições, termos, domínios, significado e uso das features.

No Módulo 10 — Governança, Segurança, Privacidade e Conformidade, controla acesso, sensibilidade, restrição de uso e finalidade das features.

No Módulo 11 — Proveniência, Linhagem e Rastreabilidade, registra origem, atributos de entrada, regras e dependências das features.

No Módulo 12 — Versionamento e Reprodutibilidade, controla versões, parâmetros, regras e ambientes usados para gerar features.

No Módulo 13 — Representações Analíticas, consome representações tabulares, temporais e em grafos para criar features específicas dessas estruturas.

6. Formalização sintética do módulo

O Módulo 14 pode ser formalizado como um processo de geração e gestão de features a partir de ativos analíticos.

Considerando o conjunto de ativos disponíveis:

A = {a₁, a₂, ..., aₙ}

e o conjunto de unidades de análise:

U = {u₁, u₂, ..., uₘ}

o módulo produz um conjunto de features:

F = {f₁, f₂, ..., fₖ}

Cada feature fᵢ pode ser descrita por:

fᵢ = {id, nome, descrição, tipo, unidade_análise, atributos_origem, regra_cálculo, parâmetros, janela_temporal, granularidade, versão, qualidade, sensibilidade, finalidade, lineage}

A matriz de features pode ser representada por:

X ∈ Rⁿˣᵖ

em que:

  • n representa o número de unidades de análise;
  • p representa o número de features;
  • Xᵢⱼ representa o valor da feature j para a unidade i.

A feature store pode ser representada como um repositório estruturado:

FS = {F, M, V, Q, G, L}

em que:

  • F representa o conjunto de features;
  • M representa metadados das features;
  • V representa versões;
  • Q representa métricas de qualidade;
  • G representa regras de governança;
  • L representa linhagem.

Dessa forma, uma feature não é apenas uma variável calculada, mas um ativo com definição, origem, regra, qualidade, versão, governança e rastreabilidade.

8. Processamento

O processamento do Módulo 14 envolve identificação, criação, validação, documentação, armazenamento, versionamento, disponibilização e monitoramento das features.

8.1 Identificação de necessidades analíticas

A primeira atividade consiste em identificar quais features são necessárias para responder a uma pergunta analítica, construir um modelo, alimentar um dashboard, gerar um indicador, compor um grafo, apoiar uma decisão ou disponibilizar um produto de dados.

Essa identificação deve considerar a finalidade, a unidade de análise, a variável-alvo, a temporalidade, as relações, os grupos relevantes e as restrições de uso.

8.2 Seleção de atributos e fontes de origem

A segunda atividade consiste em selecionar os atributos, datasets, eventos, relações ou indicadores que servirão de base para criação das features. Essa seleção deve considerar qualidade, sensibilidade, completude, rastreabilidade e adequação semântica dos atributos.

8.3 Definição das regras de criação de features

A terceira atividade consiste em definir regras formais para gerar features. Essas regras podem envolver agregações, transformações, codificações, janelas temporais, cálculos estatísticos, métricas de grafos, combinações de atributos, normalizações, discretizações ou embeddings.

Cada regra deve ser documentada com fórmula, parâmetros, atributos de origem, granularidade, unidade de análise, janela temporal e justificativa.

8.4 Construção de features tabulares

A quarta atividade consiste em construir features em formato tabular. Essas features podem incluir atributos originais, atributos derivados, indicadores, codificações categóricas, agregações por unidade de análise, normalizações e variáveis de contexto.

8.5 Construção de features temporais

A quinta atividade consiste em criar features que capturam evolução temporal, como médias móveis, tendências, variações, contagens por janela, eventos recentes, tempo desde o último evento, sequência de estados, persistência, recorrência ou sazonalidade.

8.6 Construção de features relacionais e baseadas em grafos

A sexta atividade consiste em criar features a partir de relações entre entidades. Isso pode incluir grau, grau ponderado, centralidade, PageRank, número de vizinhos, pertencimento a comunidades, distância média, componentes, densidade local, embeddings de nós ou métricas de grafos temporais.

8.7 Avaliação de qualidade das features

A sétima atividade consiste em avaliar as features quanto à completude, validade, distribuição, estabilidade, redundância, multicolinearidade, correlação, sensibilidade, risco de vazamento de informação e adequação ao uso.

Features com baixa qualidade, instabilidade ou risco elevado devem ser sinalizadas, revisadas ou excluídas.

8.8 Avaliação de vazamento de informação

A oitava atividade consiste em verificar se uma feature contém informação que não estaria disponível no momento da predição ou decisão. Esse problema, conhecido como data leakage, pode gerar resultados artificialmente elevados e comprometer a validade de modelos.

A verificação deve considerar janela de observação, janela de predição, data de disponibilidade do atributo e sequência temporal dos eventos.

8.9 Registro e documentação das features

A nona atividade consiste em registrar cada feature em um catálogo ou feature store, documentando nome, descrição, tipo, origem, regra, unidade de análise, granularidade, janela temporal, versão, qualidade, sensibilidade, limitações e consumidores.

8.10 Armazenamento e disponibilização na feature store

A décima atividade consiste em armazenar features aprovadas em uma estrutura organizada, permitindo reuso por modelos, dashboards, APIs, análises e agentes inteligentes.

A feature store deve preservar versões, metadados, lineage, status de qualidade e regras de acesso.

10.1 Taxa de Features Documentadas — TFD

A Taxa de Features Documentadas mede a proporção de features com documentação mínima.

TFD = número de features documentadas / número total de features criadas

Uma feature pode ser considerada documentada quando possui nome, descrição, origem, regra de cálculo, unidade de análise, granularidade, versão e finalidade.

10.2 Taxa de Features com Lineage — TFL

A Taxa de Features com Lineage mede a proporção de features cuja origem e regra de derivação estão registradas.

TFL = número de features com lineage / número total de features criadas

Essa métrica é essencial para auditoria, explicabilidade e reprodutibilidade.

10.3 Completude Média das Features — CMF

A Completude Média das Features avalia o grau médio de preenchimento das features.

CMF = soma das taxas de completude das features / número total de features avaliadas

Features com baixa completude podem ser inadequadas para modelos ou dashboards.

10.4 Taxa de Features com Vazamento — TFVaz

A Taxa de Features com Vazamento mede a proporção de features que apresentam risco de utilizar informação futura, indevida ou indisponível no momento da decisão.

TFVaz = número de features com risco de vazamento / número total de features avaliadas

Essa métrica é especialmente importante em problemas preditivos temporais.

10.5 Índice de Qualidade da Feature — IQF

O Índice de Qualidade da Feature pode ser definido como um indicador composto que considera completude, validade, estabilidade, rastreabilidade, documentação e ausência de vazamento.

Uma formulação possível é:

IQF(fᵢ) = w₁C + w₂V + w₃E + w₄L + w₅D + w₆(1 − Z)

em que:

  • C representa completude;
  • V representa validade;
  • E representa estabilidade;
  • L representa lineage;
  • D representa documentação;
  • Z representa risco de vazamento;
  • wᵢ representa o peso de cada dimensão.

23. Critérios gerais de validação do módulo

O Módulo 14 pode ser considerado adequadamente implementado quando:

  1. as necessidades analíticas foram identificadas;
  2. a unidade de análise das features foi definida;
  3. os atributos de origem foram documentados;
  4. as regras de cálculo foram registradas;
  5. as features possuem versionamento;
  6. a qualidade das features foi avaliada;
  7. riscos de vazamento foram verificados;
  8. features sensíveis ou proxies foram identificadas;
  9. features aprovadas foram armazenadas na feature store;
  10. features possuem lineage suficiente para auditoria;
  11. features usadas em modelos, dashboards ou APIs possuem manifesto;
  12. features críticas são monitoradas quanto a drift, estabilidade e uso.

24. Relação com o pipeline principal

O Módulo 14 se relaciona com todos os módulos do pipeline principal.

Com o Módulo 1 — Fontes e Inventário de Dados, utiliza atributos, entidades, granularidade e fontes candidatas para criação de features.

Com o Módulo 2 — Extração, Ingestão e Armazenamento Bruto, utiliza snapshots e dados brutos para garantir reprocessabilidade.

Com o Módulo 3 — Integração e Harmonização, utiliza relações entre fontes e entidades integradas para criar features combinadas.

Com o Módulo 4 — Transformação, Limpeza e Enriquecimento, utiliza dados tratados e também formaliza atributos derivados mais complexos.

Com o Módulo 5 — Qualidade e Validação de Dados, utiliza métricas de qualidade para aprovar ou rejeitar features.

Com o Módulo 6 — Desbalanceamento, Vieses e Equidade, avalia risco de viés, proxies sensíveis e impacto por grupo.

Com o Módulo 7 — Armazenamento Analítico, utiliza data marts e estruturas persistidas como base para features.

Com o Módulo 8 — Disponibilização e Consumo Analítico, fornece features para produtos, dashboards, APIs, modelos e agentes inteligentes.

25. Relação com os demais módulos transversais

O Módulo 14 depende do Módulo 9 — Metadados, Catálogo e Semântica dos Dados, pois cada feature deve possuir definição, descrição, conceito associado, domínio, finalidade e documentação semântica.

Relaciona-se com o Módulo 10 — Governança, Segurança, Privacidade e Conformidade, pois features podem ser sensíveis, restritas ou funcionar como proxies de atributos protegidos.

Relaciona-se com o Módulo 11 — Proveniência, Linhagem e Rastreabilidade, pois cada feature deve apontar para seus atributos de origem, regras, versões e processos.

Relaciona-se com o Módulo 12 — Versionamento e Reprodutibilidade, pois features devem possuir versão, parâmetros, ambiente e manifestos de execução.

Relaciona-se com o Módulo 13 — Representações Analíticas, pois features podem ser derivadas de tabelas, séries temporais, eventos, grafos e grafos temporais.

Relaciona-se com o módulo de Monitoramento, Drift e Evolução dos Ativos, pois features devem ser monitoradas quanto a estabilidade, drift, degradação, obsolescência e alteração de distribuição.

Relaciona-se com o módulo de Agentes Inteligentes de Apoio ao Pipeline, pois agentes podem consultar o catálogo de features, sugerir novas features, explicar regras de cálculo, detectar redundâncias e recomendar reuso.

26. Exemplo de aplicação no domínio educacional

Em uma aplicação educacional, o Módulo 14 pode criar features para análise de evasão, desempenho, permanência ou engajamento estudantil.

Exemplos de features tabulares incluem idade no ingresso, escola de origem, curso, turno, forma de ingresso, renda familiar, média geral, número de reprovações e frequência média.

Exemplos de features temporais incluem evolução da média por semestre, frequência acumulada nos últimos períodos, número de reprovações recentes, tempo desde a última matrícula ativa, variação de desempenho e sequência de situações acadêmicas.

Exemplos de features relacionais ou de grafos incluem número de disciplinas conectadas ao estudante, centralidade em rede de interações no AVA, grau em grafo estudante-disciplina, proximidade com estudantes de mesmo curso, participação em comunidades acadêmicas ou embeddings de nós.

Essas features poderiam ser armazenadas em uma feature store educacional, documentadas com regras, versões, qualidade, sensibilidade e riscos de uso.

27. Exemplo de aplicação em políticas públicas

Em uma aplicação em políticas públicas, o Módulo 14 pode criar features para análise de vulnerabilidade, cobertura de serviços, risco territorial ou priorização de atendimento.

Exemplos de features tabulares incluem renda média territorial, número de equipamentos públicos, taxa de atendimento, índice de escolaridade, cobertura de saúde e indicadores socioeconômicos.

Exemplos de features temporais incluem evolução mensal de atendimentos, crescimento da demanda, frequência de ocorrência por território, tendência de vulnerabilidade e sazonalidade de serviços.

Exemplos de features relacionais ou de grafos incluem centralidade de equipamentos públicos, distância relacional entre famílias e serviços, número de vínculos com políticas públicas, comunidades territoriais e persistência de atendimentos ao longo do tempo.

Essas features poderiam alimentar painéis, modelos de priorização, indicadores territoriais ou APIs de apoio à gestão pública.

28. Contribuição técnico-científica do módulo

A contribuição do Módulo 14 está em formalizar feature engineering e feature store como componentes transversais da Engenharia Analítica de Dados.

Em vez de tratar features como colunas criadas informalmente em scripts isolados, o UnespDataLens-RM propõe que elas sejam tratadas como ativos analíticos documentados, avaliados, versionados, rastreáveis, governados e reutilizáveis.

Essa formalização contribui para:

  • aumentar o valor analítico dos dados;
  • reduzir duplicidade e inconsistência de cálculos;
  • melhorar reprodutibilidade de modelos e dashboards;
  • documentar origem e regra de cada feature;
  • avaliar qualidade, estabilidade e risco de vazamento;
  • apoiar análise tabular, temporal e relacional;
  • permitir reutilização de features em diferentes produtos;
  • controlar features sensíveis ou com risco de proxy;
  • fortalecer a integração entre dados, modelos, dashboards e agentes inteligentes;
  • estruturar uma base reutilizável de atributos analíticos no ecossistema UnespDataLens-RM.

29. Síntese do módulo

O Módulo Transversal 14 — Feature Engineering e Feature Store organiza o processo de criação, documentação, validação, armazenamento e reutilização de atributos analíticos no UnespDataLens-RM. Ele transforma variáveis derivadas, indicadores, métricas temporais, métricas relacionais e embeddings em ativos formais, governados e rastreáveis.

Sua importância é transversal porque features podem ser produzidas a partir de diferentes representações e consumidas por diversos produtos, como modelos, dashboards, APIs, relatórios, grafos e agentes inteligentes.

Ao formalizar feature engineering e feature store por meio de artefatos, métricas, regras, manifestos e critérios de validação, o módulo fortalece a qualidade, a reprodutibilidade, a governança e o reuso dos atributos analíticos produzidos pelo UnespDataLens-RM.
